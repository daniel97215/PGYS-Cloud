# PGYS — Roadmap stratégique sur 18 mois

**Version :** 1.0
**Statut :** Brouillon stratégique à valider
**Période couverte :** Juillet 2026 à décembre 2027
**Nature du document :** Séquencement stratégique de l’entreprise et du produit

---

# 1. Objet du document

Cette roadmap définit l’ordre dans lequel PGYS doit avancer au cours des 18 prochains mois.

Elle ne constitue pas une liste exhaustive de fonctionnalités.

Elle vise à répondre à cinq questions :

1. Que devons-nous stabiliser en premier ?
2. Quand intégrer le salon de coiffure ?
3. Quand commencer à prospecter ?
4. Quels résultats doivent être obtenus avant d’élargir le produit ?
5. Quels sujets devons-nous volontairement reporter ?

L’objectif est d’éviter une dispersion entre :

* le client pilote 1 ;
* le client pilote 2 ;
* le salon de coiffure ;
* les futurs profils annoncés ;
* le nouveau socle SaaS ;
* la commercialisation ;
* les différents modules envisagés.

---

# 2. Principe directeur

> **Un seul chantier stratégique principal par phase.**

PGYS peut maintenir plusieurs clients et plusieurs produits existants.

Mais l’entreprise ne doit pas chercher simultanément à :

* refondre tout le logiciel ;
* construire plusieurs verticales ;
* développer toute la plateforme ;
* lancer le marketing ;
* créer un Sales OS ;
* intégrer de l’intelligence artificielle ;
* proposer une offre Cloud complète.

Chaque phase doit produire un résultat vérifiable avant le passage à la suivante.

---

# 3. Objectif à 18 mois

À la fin de la période, PGYS doit disposer de :

* un socle de gestion commun clairement défini ;
* une première verticale PGYS Services utilisable ;
* au moins trois entreprises utilisant un cycle métier comparable ;
* une offre commerciale standardisée ;
* un onboarding reproductible ;
* un pricing testé ;
* plusieurs clients payants ;
* une première méthode d’acquisition ;
* une distinction claire entre le cœur commun et les extensions verticales ;
* une décision documentée sur la verticale suivante.

L’objectif n’est pas de disposer d’un écosystème complet.

L’objectif est de prouver que PGYS peut transformer un besoin métier réel en un SaaS standardisé et commercialisable.

---

# 4. Les trois axes de travail

Toute la roadmap repose sur trois axes.

## Axe A — Produit commun

Construire progressivement le noyau partagé :

* entreprises ;
* utilisateurs ;
* tiers ;
* catalogue ;
* pièces commerciales ;
* facturation ;
* règlements ;
* documents ;
* pilotage.

## Axe B — Verticale PGYS Services

Valider une expérience adaptée aux petites entreprises de services :

* saisie rapide ;
* interface tactile ;
* commandes ou prestations ;
* réalisation ;
* planning ;
* facturation ;
* encaissement ;
* synthèse quotidienne.

## Axe C — Validation commerciale

Prouver que le produit peut être :

* compris ;
* vendu ;
* déployé ;
* utilisé ;
* renouvelé ;
* maintenu avec une marge acceptable.

Les trois axes doivent progresser ensemble, mais une phase peut en privilégier un.

---

# 5. Phase 1 — Cadrage et stabilisation des pilotes

**Période : juillet à septembre 2026**

## Objectif principal

Transformer les deux logiciels actuellement utilisés en sources de connaissance produit fiables.

Cette phase ne vise pas encore une commercialisation large.

Elle doit permettre de comprendre précisément ce qui est :

* commun ;
* vertical ;
* spécifique ;
* obsolète ;
* critique.

---

## 5.1. Client pilote 1 — Centrale d’achat

Le client 1 doit être stabilisé comme pilote de profondeur métier.

Travaux prioritaires :

* documenter le cycle réel de réception ;
* documenter les expéditions ;
* formaliser le retour des quantités reçues ;
* formaliser la création des pièces client ;
* formaliser la rémunération du producteur ;
* clarifier la notion de commission ;
* documenter les avoirs ;
* documenter les règlements entrants et sortants ;
* fiabiliser l’export comptable ;
* identifier les fonctions propres à cette activité.

Le but n’est pas de migrer immédiatement ce client vers PGYS Services.

Le but est d’extraire les capacités génériques utiles au socle commun.

---

## 5.2. Client pilote 2 — Entreprise artisanale

Le client 2 devient la référence principale de PGYS Services.

Travaux prioritaires :

* documenter le cycle commande-réalisation-facture-règlement ;
* observer l’utilisation réelle de l’écran tactile ;
* identifier les actions les plus fréquentes ;
* mesurer les étapes inutiles ;
* clarifier les statuts de commande ;
* formaliser les règlements multiples ;
* fiabiliser le récapitulatif des ventes ;
* identifier les données nécessaires au pilotage quotidien ;
* distinguer les habitudes du client des besoins génériques.

---

## 5.3. Travail stratégique commun

Pendant cette phase, PGYS doit produire :

* le glossaire métier commun ;
* la cartographie des pièces ;
* la cartographie des statuts ;
* la liste des rôles de tiers ;
* la liste des moyens de paiement ;
* la matrice cœur / configuration / verticale / spécifique ;
* les premiers indicateurs de valeur client.

---

## Critères de sortie de phase

La phase est terminée lorsque :

1. les deux cycles métier sont documentés ;
2. le noyau commun est explicitement identifié ;
3. les fonctions spécifiques sont isolées ;
4. le périmètre PGYS Services est confirmé ;
5. les principaux irritants des utilisateurs sont connus ;
6. aucun développement majeur n’est lancé sans rattachement à une catégorie claire.

---

# 6. Phase 2 — Construction du noyau PGYS Services

**Période : octobre à décembre 2026**

## Objectif principal

Construire ou consolider le parcours central de PGYS Services.

Le cycle prioritaire est :

```text id="2n8clq"
Client
→ commande ou prestation
→ réalisation
→ facture
→ règlement
→ synthèse quotidienne
```

---

## 6.1. Périmètre produit prioritaire

Le noyau doit couvrir :

* entreprise ;
* utilisateurs ;
* clients ;
* articles ;
* prestations ;
* familles ;
* commande ;
* lignes de commande ;
* commentaires et opérations ;
* prix modifiable selon autorisation ;
* remises ;
* facturation ;
* avoirs ;
* règlements multiples ;
* historique ;
* synthèse quotidienne ;
* sauvegarde ;
* export simple.

---

## 6.2. Périmètre explicitement reporté

Ne doivent pas devenir prioritaires pendant cette phase :

* CRM marketing ;
* fidélité avancée ;
* réservation en ligne ;
* portail client ;
* stock avancé ;
* achats complets ;
* production avancée ;
* automatisations complexes ;
* intelligence artificielle ;
* comptabilité complète ;
* marketplace ;
* plusieurs verticales.

---

## 6.3. Préparation du salon de coiffure

Le salon ne doit pas déclencher immédiatement une refonte du produit.

Avant son intégration, PGYS doit documenter :

* nombre de collaborateurs ;
* types de prestations ;
* durées ;
* horaires ;
* prise de rendez-vous ;
* ventes de produits ;
* moyens de paiement ;
* gestion des annulations ;
* besoins de fidélité ;
* besoins de stock ;
* besoins de caisse ;
* outils actuels.

Le planning doit être conçu à partir du cas réel, mais comme une capacité générique de rendez-vous et de ressources.

---

## Critères de sortie de phase

La phase est terminée lorsque :

1. le cycle principal fonctionne sans double saisie majeure ;
2. les règlements multiples sont fiables ;
3. la synthèse quotidienne est exploitable ;
4. le produit peut être démontré en moins de 20 minutes ;
5. le client pilote 2 utilise les fonctions principales au quotidien ;
6. le périmètre du planning est défini ;
7. le salon de coiffure peut être onboardé sans créer un second produit.

---

# 7. Phase 3 — Intégration du salon de coiffure

**Période : janvier à mars 2027**

## Objectif principal

Utiliser le salon de coiffure pour valider que PGYS Services peut servir un deuxième métier proche avec le même socle.

Le salon est un test de réutilisabilité.

Ce n’est pas uniquement un nouveau client.

---

## 7.1. Capacités à valider

Le salon doit permettre de valider :

* planning journalier et hebdomadaire ;
* rendez-vous ;
* collaborateurs ;
* durées de prestation ;
* affectation d’une prestation ;
* vente de produits ;
* facturation immédiate ;
* règlements ;
* historique client ;
* activité quotidienne.

---

## 7.2. Restrictions de périmètre

Les éléments suivants doivent être reportés sauf nécessité absolue :

* réservation en ligne ;
* programme de fidélité complexe ;
* cartes cadeaux ;
* forfaits avancés ;
* commissions salariales ;
* paie ;
* campagnes marketing ;
* gestion complète des stocks ;
* synchronisation avec plusieurs plateformes externes.

---

## 7.3. Objectif stratégique

À la fin de cette phase, PGYS doit pouvoir démontrer que :

* le client artisan et le salon utilisent le même cœur ;
* les différences sont gérées par configuration ou modules ;
* le planning ne fragilise pas le produit ;
* le catalogue gère produits et prestations ;
* l’interface tactile reste pertinente dans les deux cas.

---

## Critères de sortie de phase

La phase est terminée lorsque :

1. le salon utilise le produit dans son activité réelle ;
2. le planning est suffisamment fiable ;
3. le produit gère deux métiers proches sans duplication ;
4. l’onboarding est documenté ;
5. le temps de formation est mesuré ;
6. les demandes spécifiques du salon sont identifiées ;
7. la promesse commerciale PGYS Services est confirmée ou ajustée.

---

# 8. Phase 4 — Standardisation de l’offre

**Période : avril à juin 2027**

## Objectif principal

Passer d’un produit utilisé par des pilotes à une offre vendable à un client inconnu.

Cette phase porte autant sur l’offre que sur le logiciel.

---

## 8.1. Livrables commerciaux

PGYS doit disposer de :

* une proposition de valeur claire ;
* une présentation courte ;
* deux scénarios de démonstration ;
* une grille tarifaire testable ;
* une fiche d’offre ;
* une procédure de qualification ;
* une procédure d’onboarding ;
* une liste des exclusions ;
* un contrat ou des conditions de service adaptés ;
* une méthode de migration standard.

---

## 8.2. Livrables produit

Le produit doit disposer de :

* profils de configuration ;
* données de démonstration ;
* modèles de documents ;
* rôles standards ;
* paramètres de base ;
* procédure de sauvegarde ;
* procédure de restauration ;
* documentation minimale ;
* suivi des incidents ;
* suivi des demandes clients.

---

## 8.3. Offre pilote externe

PGYS peut recruter un troisième client comparable hors du cercle initial.

Le troisième client doit être choisi pour sa proximité avec le cœur validé.

Il ne doit pas être sélectionné uniquement parce qu’il accepte de signer.

Critères recommandés :

* 1 à 10 utilisateurs ;
* cycle proche du client 2 ou du salon ;
* décisionnaire accessible ;
* besoin réel ;
* outils dispersés ;
* volonté de tester ;
* faible niveau d’intégration externe ;
* budget disponible.

---

## Critères de sortie de phase

La phase est terminée lorsque :

1. l’offre est compréhensible sans longue explication ;
2. le prix peut être présenté clairement ;
3. l’onboarding suit une procédure ;
4. le troisième client utilise le produit ;
5. le déploiement ne nécessite pas de refonte ;
6. le coût d’accompagnement est mesuré ;
7. les premiers témoignages peuvent être obtenus.

---

# 9. Phase 5 — Première commercialisation structurée

**Période : juillet à septembre 2027**

## Objectif principal

Valider une méthode simple et reproductible pour trouver et convertir des prospects.

La priorité n’est pas le volume.

La priorité est d’apprendre quel segment répond le mieux.

---

## 9.1. Segments de test

PGYS peut tester deux sous-segments maximum.

Exemples :

* ateliers et artisans avec accueil client ;
* salons et services sur rendez-vous.

PGYS ne doit pas prospecter simultanément :

* artisans ;
* salons ;
* garages ;
* associations ;
* commerces ;
* agriculteurs ;
* centrales d’achat.

La comparaison doit rester lisible.

---

## 9.2. Canal commercial initial

Le canal recommandé est une prospection directe et ciblée :

* réseau local ;
* recommandations ;
* email personnalisé ;
* téléphone ;
* démonstration ;
* partenariats locaux.

La publicité payante n’est pas prioritaire tant que :

* le message n’est pas validé ;
* le taux de conversion n’est pas connu ;
* l’onboarding n’est pas stabilisé.

---

## 9.3. Objectifs de la phase

* contacter un volume limité de prospects qualifiés ;
* obtenir des rendez-vous ;
* mesurer les objections ;
* tester le pricing ;
* signer quelques clients ;
* identifier le sous-segment le plus réceptif.

---

## Critères de sortie de phase

La phase est terminée lorsque :

1. PGYS sait quel message génère de l’intérêt ;
2. plusieurs démonstrations sont réalisées ;
3. au moins quelques prospects acceptent une proposition ;
4. au moins un nouveau client paie le prix standard ;
5. les objections sont documentées ;
6. le segment le plus prometteur est identifiable ;
7. le cycle de vente moyen est connu.

---

# 10. Phase 6 — Validation économique et choix de la suite

**Période : octobre à décembre 2027**

## Objectif principal

Décider si PGYS Services mérite une accélération, un recentrage ou une révision.

Cette décision doit être fondée sur des données.

---

## 10.1. Questions à trancher

* Les clients utilisent-ils réellement le produit ?
* Le pricing est-il accepté ?
* Le support reste-t-il maîtrisé ?
* Le salon et l’artisan partagent-ils réellement le même cœur ?
* Quel sous-segment convertit le mieux ?
* Quel sous-segment reste le plus longtemps ?
* Quelle fonction crée le plus de valeur ?
* Quelle fonction génère le plus de complexité ?
* Le produit peut-il être déployé sans développement spécifique ?
* PGYS doit-il approfondir cette verticale ou préparer la suivante ?

---

## 10.2. Décisions possibles

### Option A — Accélération

Choisie si :

* le marché répond ;
* les clients paient ;
* l’usage est régulier ;
* la marge est acceptable ;
* les demandes restent proches.

Conséquences :

* acquisition plus structurée ;
* recrutement ou sous-traitance ciblée ;
* amélioration de l’onboarding ;
* extensions prioritaires ;
* éventuellement réservation en ligne ou portail client.

### Option B — Recentrage

Choisie si un sous-segment fonctionne mieux que les autres.

Exemple :

* ateliers oui ;
* salons non.

Conséquence :

* spécialisation plus forte ;
* message plus précis ;
* suppression ou report de fonctions secondaires.

### Option C — Révision

Choisie si :

* l’usage est faible ;
* le prix accepté est trop bas ;
* le support est trop élevé ;
* les besoins divergent trop fortement.

Conséquence :

* revoir la promesse ;
* revoir le segment ;
* réduire le périmètre ;
* éviter d’investir davantage sans preuve.

# 12. Ce qui ne doit pas être lancé pendant les 18 mois

Sauf opportunité exceptionnelle clairement rentable, PGYS ne doit pas lancer :

* une deuxième grande verticale commerciale ;
* un ERP généraliste ;
* une suite RH ;
* une comptabilité complète ;
* un logiciel de paie ;
* un Sales OS commercialisé séparément ;
* une marketplace ;
* une suite collaborative complète ;
* une messagerie PGYS ;
* une GED avancée ;
* une intelligence artificielle autonome ;
* une infrastructure cloud vendue comme produit principal ;
* une application mobile complète distincte ;
* une refonte totale des deux logiciels historiques.

Ces sujets peuvent être étudiés ou prototypés, mais ils ne doivent pas détourner l’entreprise de la validation de PGYS Services.

---

# 13. Place de la centrale d’achat dans la roadmap

Le client 1 doit continuer à être maintenu.

Toutefois, la verticale centrale d’achat ne doit pas devenir le principal axe commercial avant la validation de PGYS Services.

Pendant les 18 mois :

* les bugs sont corrigés ;
* les besoins critiques sont traités ;
* les règles métier sont documentées ;
* les composants génériques sont identifiés ;
* aucune refonte commerciale majeure n’est lancée.

À la fin de la période, PGYS pourra décider si cette activité justifie une verticale :

* PGYS Centrale ;
* PGYS Négoce ;
* PGYS Intermédiation.

Cette décision dépendra de la taille du marché et de l’existence d’autres prospects comparables.

---

# 14. Gestion des nouveaux profils prévus

Les nouveaux profils annoncés ne doivent pas automatiquement modifier la roadmap.

Chaque opportunité sera évaluée selon cinq critères :

1. proximité avec PGYS Services ;
2. potentiel de réutilisation ;
3. revenu potentiel ;
4. coût d’intégration ;
5. impact sur la roadmap.

## Règle de décision

### Cas proche de PGYS Services

Le client peut être accepté comme nouveau pilote ou client standard.

### Cas proche de la centrale d’achat

Le client peut renforcer l’hypothèse de la seconde verticale.

### Cas totalement différent

Le client peut être refusé, reporté ou traité comme prestation séparée sans modifier le produit principal.

---

# 15. Allocation recommandée des ressources

Comme PGYS repose encore largement sur une seule personne, le temps doit être protégé.

Répartition indicative :

* **60 % :** cœur PGYS Services ;
* **20 % :** maintien et support des clients existants ;
* **10 % :** documentation et onboarding ;
* **10 % :** commercial et recherche marché.

Pendant la phase de prospection, la répartition pourra évoluer :

* **45 % :** produit ;
* **20 % :** support ;
* **20 % :** commercial ;
* **15 % :** documentation et opérations.

Le développement ne doit pas absorber 100 % du temps disponible.

# 17. Jalons stratégiques

## Jalon 1 — Noyau confirmé

Le cœur commun est documenté et accepté.

## Jalon 2 — Deuxième métier intégré

Le salon utilise le même socle que le client 2.

## Jalon 3 — Offre vendable

Un prospect externe peut comprendre, acheter et utiliser l’offre.

## Jalon 4 — Prix validé

Un client paie le tarif standard.

## Jalon 5 — Déploiement reproductible

Un nouveau client peut être installé sans développement spécifique majeur.

## Jalon 6 — Verticale validée

Plusieurs entreprises proches utilisent et renouvellent le produit.

Aucune phase suivante ne doit être considérée comme acquise tant que le jalon correspondant n’est pas atteint.

---

# 18. Risques majeurs

## 18.1. Réécriture permanente

Vouloir tout reconstruire proprement avant de vendre peut retarder indéfiniment la validation.

## 18.2. Nouveaux clients trop différents

Chaque nouveau profil peut détourner le produit de son cœur.

## 18.3. Maintenance des solutions historiques

Le support des logiciels existants peut consommer la majorité du temps disponible.

## 18.4. Sous-tarification

Des prix trop bas empêcheraient de financer l’accompagnement.

## 18.5. Surdimensionnement du salon

Le planning, la fidélité et la réservation peuvent élargir excessivement le périmètre.

## 18.6. Absence de prospection

Un produit utilisé par deux clients ne constitue pas encore un marché validé.

---

# 19. Règles d’arbitrage

Lorsqu’une nouvelle idée apparaît, elle doit être évaluée dans cet ordre :

1. Est-elle nécessaire au jalon actuel ?
2. Est-elle utile à plusieurs clients ?
3. Renforce-t-elle PGYS Services ?
4. Peut-elle être reportée sans risque ?
5. Génère-t-elle un revenu ou une preuve de marché ?
6. Quel chantier sera retardé si elle est acceptée ?

Une bonne idée peut être refusée si elle arrive au mauvais moment.

---

# 20. Décisions actées

* PGYS Services constitue la priorité stratégique des 18 prochains mois.
* Le client 2 est le pilote principal de reproductibilité.
* Le salon de coiffure sert de validation d’un deuxième métier.
* Le client 1 reste le pilote de profondeur métier.
* Le cœur commun doit être stabilisé avant toute multiplication de verticales.
* Le planning est prioritaire pour intégrer le salon, mais son périmètre doit rester simple.
* La commercialisation structurée commence après la validation du salon et de l’onboarding.
* La première prospection vise au maximum deux sous-segments.
* Les nouveaux profils ne modifient pas automatiquement la roadmap.
* La deuxième verticale ne sera décidée qu’après validation économique de PGYS Services.
* L’intelligence artificielle et les grands modules périphériques sont reportés.
* Chaque phase doit produire un résultat mesurable.

---

# 21. Résultat attendu à fin 2027

À la fin de décembre 2027, PGYS doit être capable de déclarer :

> **Nous disposons d’un cœur de gestion commun, d’une première verticale validée auprès de plusieurs entreprises, d’une offre claire, d’un prix accepté et d’une méthode reproductible pour transformer un prospect en client actif.**

Si ce résultat est atteint, PGYS aura franchi le passage le plus difficile :

> passer d’un logiciel développé pour quelques clients à un véritable produit SaaS commercialisable.

---

# 22. Principe final

> **La roadmap n’a pas pour fonction de prévoir tout ce que PGYS fera. Elle sert à protéger ce que PGYS doit réussir en premier.**
